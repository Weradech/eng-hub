> **TL;DR** — PM2 has three non-obvious failure modes that look unrelated: killing the wrong process destroys unrelated services, `restart` silently caches stale env vars, and a Node.js upgrade can flip the native module ABI mid-run. All three have simple fixes once you know the pattern.

---

## Gotcha 1: The God Daemon Is Not a Zombie

When you run `pm2 start` for the first time, PM2 launches a background daemon (`pm2-runtime` or `God daemon`). Every app you start is a child of that daemon.

If you kill `pm2-runtime` thinking it's a zombie process, you kill every app attached to it — not just the one you wanted to stop.

```bash
# You see this in ps aux and want to kill "the stuck thing":
pm2-runtime /home/ball/syntech_crm/src/server.js   ← THIS IS THE DAEMON
pm2-runtime /home/ball/jig-api/src/index.js
pm2-runtime /home/ball/syntech_crm/src/server.js

# Killing the first line kills all three.
```

**The safe check before any kill:**

```bash
pm2 list          # see all managed apps and their pm2 IDs
pm2 stop <name>   # stop by name, not by killing the OS process
pm2 delete <name> # remove from pm2's process table
```

Never use `kill $(pgrep pm2-runtime)`. Use `pm2 stop <name>` or `pm2 kill` (shuts down the entire daemon cleanly, all apps stop gracefully).

## Gotcha 2: `pm2 restart` Does Not Refresh `.env`

PM2 caches environment variables at the time an app is first started. `pm2 restart <name>` — and even `pm2 restart <name> --update-env` — does not reliably pick up changes to the `.env` file.

```bash
# You changed DB_HOST in .env
pm2 restart my-app            # ← old DB_HOST still in memory
pm2 restart my-app --update-env  # ← still unreliable in practice
```

**The only reliable pattern:**

```bash
pm2 delete my-app
pm2 start ecosystem.config.js --only my-app
```

If you don't have an ecosystem file, pass the env inline:

```bash
pm2 delete my-app
pm2 start src/index.js --name my-app --env production
```

It feels like extra work, but `delete + start` is idempotent and predictable. `restart` is not.

## Gotcha 3: Native Module ABI Flip-Flop After Node Upgrade

If your app uses a native Node addon (e.g., `better-sqlite3`), the compiled `.node` binary is tied to a specific Node ABI version. When Node.js is upgraded, the ABI number changes.

On a server managed with `nvm` or a system package manager, two Node versions can coexist. PM2's daemon is launched by one version; if the system default Node flips to another, the daemon may restart workers using the new ABI — crashing with `exit code 139` (segfault) or `Error: NODE_MODULE_VERSION mismatch`.

```
node v20 (ABI 115) starts pm2 daemon
  └─ better-sqlite3 compiled for ABI 115 ✓

system upgrade → node v18 becomes default (ABI 108)
  └─ pm2 daemon restarts under v18
  └─ better-sqlite3 still compiled for ABI 115 → segfault on load
```

**The durable fix — pin the interpreter in `ecosystem.config.js`:**

```js
module.exports = {
  apps: [{
    name: 'my-app',
    script: 'src/index.js',
    interpreter: '/usr/bin/node',   // absolute path, not 'node'
    interpreter_args: '',
    env_production: {
      NODE_ENV: 'production'
    }
  }]
}
```

Using an absolute path for `interpreter` bypasses whatever `node` the shell resolves at daemon-restart time. The app always runs under the same binary.

If you've already hit the ABI mismatch, the recovery is:

```bash
# 1. Find which node version the app should use
which node && node --version

# 2. Rebuild the native module for the target version
npm rebuild better-sqlite3

# 3. Restart with the pinned interpreter
pm2 delete my-app
pm2 start ecosystem.config.js --only my-app --env production
```

## Summary

| Symptom | Root Cause | Fix |
|---------|-----------|-----|
| Unrelated apps die unexpectedly | Killed `pm2-runtime` daemon | Use `pm2 stop/delete` by name |
| Env change not picked up after restart | `pm2 restart` caches env | `pm2 delete` + `pm2 start` |
| `exit 139` / ABI mismatch after node update | Native addon compiled for wrong ABI | Pin `interpreter` in ecosystem config + `npm rebuild` |
