> **TL;DR** — You can pull structured fields out of supplier PDFs — PO numbers, line items, GR references — without sending a single page to a cloud LLM. Try the embedded text layer first (`pdfplumber`), fall back to OCR (`Tesseract`) only for scanned pages, and extract fields with layout anchors instead of fuzzy matching. The result is cheaper, deterministic, offline, and keeps supplier documents on your own infrastructure.

---

## Why not just ship it to a cloud model?

The fastest-to-demo approach is to hand each PDF to a hosted vision-LLM and ask for JSON. For an internal document pipeline that processes invoices, goods-receipt notes, and POs, that convenience carries costs that compound:

- **Data residency.** Supplier invoices contain prices, terms, and contacts. Routing every document through a third-party API means your commercial data leaves the building on every run.
- **Per-document cost.** A pipeline handling a steady stream of documents pays per page, forever. A local pipeline pays once for the hardware it already has.
- **Determinism.** The same invoice should extract the same fields every time. A model that can be silently updated upstream can change its output on a document that worked yesterday — with no diff to point at.
- **Offline + latency.** A receiving-dock workflow shouldn't block on an external API being reachable and fast.

None of this means "never use an LLM." It means **don't reach for the heaviest, least controllable tool first** when most of the documents are structured PDFs you can parse directly. (This is the same instinct behind [the co-master sunset vision]({% post_url 2026-06-29-wms-odoo-comaster %}) — don't hardwire a dependency you'd struggle to remove.)

## The pipeline: text-layer first, OCR as fallback

Most business PDFs — anything generated by an ERP, an accounting package, or Word — carry a real embedded text layer. You don't need to OCR those; you can read the text and its coordinates directly. OCR is only for the scanned and photographed pages.

```
for each page:
    text = pdfplumber.extract()        # embedded text layer
    if text is meaningful:
        → parse fields from text + layout   (fast, exact)
    else:                               # scanned / image-only page
        image = render(page)
        text  = tesseract(image)            # OCR fallback (slower, fuzzier)
        → parse fields from OCR text
```

Why this ordering matters: the text layer is **exact** — the characters are the real characters, and you get their `(x, y)` positions for free. OCR is an *estimate* and introduces character errors (`O`↔`0`, `l`↔`1`). Running OCR on a page that already has a clean text layer throws away accuracy and time for nothing. Branch on whether the embedded text is meaningful, and only pay for OCR when there's no alternative.

> Reach for OCR only when there's no text layer to read. A huge share of "OCR is inaccurate" pain is self-inflicted — OCR-ing pages that already contained perfectly good embedded text. Detect and skip those first.

## Extract with layout anchors, not freeform guessing

The part that makes or breaks accuracy isn't the OCR engine — it's *how you locate a field once you have the text*. Two strategies, and one is far more robust:

**Fragile:** scan the whole page text for "something that looks like a PO number." On a document with a PO number, an order number, and a quote number — all similar formats — this picks the wrong one regularly.

**Robust:** anchor on the document's *structure*. The PO number is the text immediately to the right of the label `PO No.`; the line-item quantities live in the column under the `QTY` header. `pdfplumber` gives you word coordinates and table structure, so you can say "the value in the cell to the right of this label" instead of "any string matching this regex, somewhere."

```
# anchored, not freeform:
po_no = value_right_of_label("PO No.")          # uses x/y position
lines = table_under_header(["Item", "Qty", "Unit Price"])
```

Anchoring to layout means a second PO-shaped number elsewhere on the page can't hijack the field, and it survives small wording changes far better than a page-wide regex.

## Cross-reference the document IDs

Receiving documents rarely live alone — a single physical page often carries several identifiers that tie systems together: the PO number, the GR/receipt number, the invoice number, the delivery-order number. Extracting them is only half the job; the value is in **linking** them so a PO can be traced to its goods receipt and its payment.

Pull every ID present on the page, then reconcile them against the systems that own each one — match the PO against the ERP, the GR against the warehouse record — so the extracted document slots into the existing chain instead of becoming an island. (This is the data backbone behind [going paperless on GR→IQC]({% post_url 2026-06-29-paperless-gr-iqc-3phase %}).)

## When you *do* reach for an LLM

A local pipeline isn't dogma. The pragmatic split:

- **Structured, templated PDFs** (ERP/accounting output) → text layer + layout anchors. Most of your volume.
- **Scanned, clean documents** → Tesseract + anchored extraction.
- **Genuinely messy, free-form, or handwritten** documents that defeat both → *then* an LLM earns its place, ideally on just the residual that the deterministic path couldn't handle.

Use the cheap, deterministic, on-prem path for the 90% it handles well, and escalate only the long tail. That keeps cost and data exposure proportional to the actual difficulty, instead of paying cloud-AI prices for documents a regex-with-coordinates would nail.

---

## The portable checklist

- **Text layer before OCR.** Detect embedded text and skip OCR when it exists.
- **Anchor fields to layout** (labels, columns, cell positions), not to page-wide pattern matching.
- **Pull and cross-reference every document ID** so extraction feeds the existing PO→GR→payment chain.
- **Keep it on-prem and deterministic** for the bulk; escalate only the genuinely messy tail to a heavier model.
- **Treat OCR output as estimated** — validate IDs against the systems that own them rather than trusting the characters.
